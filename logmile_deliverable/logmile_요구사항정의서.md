# 요구사항 정의서

## logmile - 화물차 운전자 피로도 실시간 모니터링 플랫폼

- 프로젝트명: `logmile`
- 버전: v5.3
- 기준 산출물: `AGENTS.md`, `logmile_infra/db/init.sql`, `logmile_infra/db/seed.sql`, 실제 BE/FE/AI/SIM/DELIVERABLE 구현물
- 작성 기준일: 2026.05.26
- 변경 기준: 시뮬레이터 3열 수평 그리드 개편, 시나리오 자동 일괄 시퀀서(오토 데모) 구동, localStorage 기반 다중 이력 관리 복구, 최상위/업체 관리자 승인 권한 흐름 반영
- 플랫폼: 웹 브라우저 기반 SPA
- 주요 사용자: 관제 관리자, 시스템 관리자

## 1. 프로젝트 개요

`logmile`은 화물차 운전자의 피로 위험을 운행 데이터 기반으로 실시간 감지하고, 관제 관리자가 차량별 피로도 상태와 판단 근거를 확인할 수 있도록 지원하는 웹 기반 관제 플랫폼이다.

피로도 판단은 연속 운행 시간, 휴식 여부, 일일 총 운행 시간, 야간 운행 시간을 기준으로 산정하며, 결과는 정상, 주의, 위험 등급으로 표시한다.

## 2. 사용자 및 권한

| 사용자 | 권한 | 주요 기능 |
|---|---|---|
| 최상위 관리자 | ROLE_SUPER_ADMIN | 업체 목록 조회, 업체 활성/비활성, 가입 관리자 승인/거절/정지/해제, 대시보드/이력 조회 |
| 업체 관리자 | ROLE_ADMIN | 소속 업체의 대시보드 조회, 차량/운전자 관리, 시뮬레이션 실행, 운행 이력/피로도 통계 조회 |

관리자 계정 상태는 `PENDING`, `ACTIVE`, `INACTIVE`, `REJECTED`, `SUSPENDED`로 구분한다. 로그인은 `ACTIVE` 상태 계정만 허용하며, 그 외 상태는 서버가 계정 상태별 오류를 반환한다.

### 4.2.1 시연용 A/B/C 시나리오 기준

시뮬레이션 기능은 실제 운영 기능의 구조를 변경하지 않고, 실제 환경에서 자동 수집되는 번호판 인식, GPS 수신, CCTV 관측 데이터를 시연 환경에서 직접 입력하여 재현하는 기능이다.

| 시나리오 | 목적 | 운행/이벤트 기준 | 예상 등급 |
|---|---|---|---|
| A 정상 | 충분 휴식이 포함된 정상 운행 | 주간 운행, 충분 휴식 30분 이상, 휴식 누락 없음, 번호판 관측 이벤트 포함 | NORMAL |
| B 주의 | 짧은 휴식과 야간 일부가 포함된 주의 운행 | 120분 이상 운행, 15분 미만 휴식, 야간 운행 일부, 휴식 누락 1회 | CAUTION |
| C 위험 | 장시간 연속 운행과 야간 장시간이 포함된 위험 운행 | 240분 이상 연속 운행, 15분 미만 휴식, 야간 2시간 이상, 휴식 누락 2회 이상 | DANGER |

시연용 위치명은 화면 또는 메모성 텍스트로 표현하고, 저장 기준은 기존 `PlateLocationType` 값인 `HIGHWAY_GATE`, `REST_AREA`, `CCTV`만 사용한다.

### 4.7 시연용 직접 입력 원칙

| 구분 | 실제 운영 흐름 | 시연용 대체 흐름 |
|---|---|---|
| 운행 시작 | 출발지 번호판 인식 및 차량/운전자 매칭으로 자동 시작 | 번호판 사진 또는 직접 입력, 시작 위치/시간 입력 후 `drive_log` 생성 |
| 운행 종료 | 도착지 번호판 인식으로 자동 종료 | 종료 위치/시간 입력 후 운행 종료 처리 |
| 입출차 기록 | 항만, 허브, 차고지, 도로 CCTV 관측으로 자동 기록 | 위치/시간/번호판을 직접 입력해 기존 `plate_event`에 저장 |
| 휴식 확인 | 휴게소 CCTV 또는 GPS 정차 패턴으로 자동 판단 | 휴식 위치와 시작/종료 시간을 직접 입력해 기존 `rest_event`에 반영 |
| 운행 시간 | GPS 누적값으로 연속/일일/야간 운행 시간 자동 계산 | 시연 프리셋 또는 직접 입력값으로 피로도 계산 결과를 확인 |

시연용 입력은 별도 핵심 테이블을 추가하지 않고 기존 `drive_log`, `gps_data`, `rest_event`, `fatigue_event`, `plate_event` 흐름을 재현하는 보조 입력으로 취급한다.

## 11. 인수 기준

| 항목 | 기준 |
|---|---|
| 번호판 인식 | 출발/도착 번호판 인식 API가 동작하고, 실패 시 수동 입력 fallback이 가능해야 한다. |
| GPS 시뮬레이션 | A/B/C 시나리오 데이터가 생성되고 백엔드로 전송되어야 한다. |
| 시연용 시뮬레이션 | 페이지에서 A/B/C 프리셋을 선택하고 실행하면 오늘 일자 기준으로 주/야간 오토 데모 시퀀스가 에러 없이 구동되고, 실시간 라이브 차트 및 이벤트 로그에 안전하게 반영되어야 한다. |
| 다중 시뮬레이션 복원 | 운행 기록 중 또는 종료 시점에 시뮬레이션 로그에 자동 적재되며, 과거 이력 클릭 시 차트와 누적 이벤트 로그가 해당 시점의 온전한 데이터로 실시간 복원되어야 한다. |
| 피로도 계산 | 운행, 휴식, 야간 운행 기준으로 정상/주의/위험 등급이 산정되어야 한다. |
| 대시보드 | 차량 목록, 피로도 점수, 등급, 판단 근거가 화면에 표시되어야 한다. |
| 관리 기능 | 차량, 운전자, 피로도 임계값을 관리할 수 있어야 한다. |
| 이력/통계 | 운행 이력과 피로도 통계를 조회할 수 있어야 한다. |
| 인증 | 관리자 로그인 후 보호된 기능에 접근할 수 있어야 한다. |
| 승인 흐름 | 신규 업체 관리자 회원가입 후 최상위 관리자의 승인 전까지 로그인 또는 보호 기능 접근이 제한되어야 한다. |
| 업체 격리 | 업체 관리자는 자기 업체의 차량, 운전자, 운행 이력만 조회 및 관리할 수 있어야 한다. |
| 통합 시연 | 데모 계정으로 로그인 후 승인, 시뮬레이션, GPS 수신, 피로도 산정, 대시보드/이력/통계 확인 흐름을 설명할 수 있어야 한다. |
