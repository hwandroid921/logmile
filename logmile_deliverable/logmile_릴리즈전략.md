# logmile 릴리즈 전략

## 화물차 운전자 피로도 실시간 모니터링 플랫폼

- 프로젝트명: `logmile`
- 작성 기준일: 2026.04.30
- 개발 기간: 2026.04.28 ~ 2026.05.27 제출 / 2026.06.02 발표
- 적용 범위: 프로젝트 시작, 기능 개발, 통합, QA, 제출, 발표까지의 전체 릴리즈 관리
- 기준 브랜치:
  - `main`: 최종 릴리즈 브랜치
  - `dev`: 통합 개발 브랜치
  - `feature/*`: 기능 개발 브랜치
  - `release/*`: 릴리즈 검증 브랜치

## 1. 릴리즈 전략 개요

`logmile`은 학습 및 연구용 프로젝트이므로 운영 배포보다 제출과 발표 시연의 안정성을 우선한다.

따라서 릴리즈 전략은 복잡한 운영 배포형 방식이 아니라, `dev`에 기능을 통합한 뒤 마일스톤마다 `release/*` 브랜치를 생성하고 검증 후 `main`에 반영하는 방식으로 운영한다.

### 기본 흐름

```text
feature/* → dev → release/vX.X.X → main → tag
```

### 브랜치별 역할

| 브랜치 | 역할 | 직접 커밋 여부 |
|---|---|---|
| `main` | 최종 제출/발표 버전 보관 | 금지 |
| `dev` | 기능 통합 및 내부 테스트 | 제한적 허용 |
| `feature/*` | 기능 단위 개발 | 허용 |
| `release/*` | QA, 버그 수정, 문서 정리 | 릴리즈 안정화 수정만 허용 |
| `hotfix/*` | 제출/발표 직전 긴급 수정 | 필요 시 허용 |

## 2. 버전 관리 규칙

버전은 Semantic Versioning 형식을 따른다.

```text
vMAJOR.MINOR.PATCH
```

| 구분 | 증가 기준 | 예시 |
|---|---|---|
| MAJOR | 제출 가능한 전체 기능 통합 완료 | `v1.0.0` |
| MINOR | 기능 묶음 또는 주요 마일스톤 완료 | `v0.2.0` |
| PATCH | 버그 수정, 문서 보완, 발표용 보정 | `v1.0.1` |

### 본 프로젝트 버전 기준

| 버전 | 의미 |
|---|---|
| `v0.1.0` | 프로젝트 기반 구조 통합 |
| `v0.1.1` | 인증/회원가입/승인 구조 통합 |
| `v0.2.0` | 백엔드 핵심 기능 통합 |

| `v1.0.0` | 최종 제출 버전 |

## 5. 릴리즈 브랜치 생성 절차

릴리즈 브랜치는 해당 마일스톤 기능이 `dev`에 모두 병합된 뒤 생성한다.

```bash
git switch dev
git pull origin dev

git switch -c release/v0.1.0
git push -u origin release/v0.1.0
```

릴리즈 브랜치에서는 신규 기능을 추가하지 않는다.

허용되는 작업은 다음으로 제한한다.

```text
- 빌드 오류 수정
- 실행 오류 수정
- 설정 누락 수정
- 문서 최신화
- seed 데이터 보정
- 테스트 실패 수정
- 제출/발표에 필요한 최소 UI 문구 수정
```

## 8. 릴리즈 검증 기준

릴리즈 검증은 코드, 데이터, 문서, 실행 환경을 모두 포함한다.

## 8.1 공통 검증

| 구분 | 검증 항목 |
|---|---|
| Git | 기준 브랜치가 맞는지 확인 |
| Git | 불필요한 untracked 파일이 없는지 확인 |
| Git | `.env`, 모델 가중치, 빌드 산출물이 포함되지 않았는지 확인 |
| 문서 | UTF-8 기준으로 한글 문서가 정상 열람되는지 확인 |
| 문서 | 산출물의 날짜, 버전, 파일명 참조가 맞는지 확인 |

## 8.2 Backend 검증

| 구분 | 검증 항목 |
|---|---|
| Build | Spring Boot 빌드 성공 |
| Auth | JWT 로그인 및 권한 검증 |
| DB | JPA Entity와 실제 테이블 구조 일치 |
| API | 차량/운전자/GPS/피로도 API 정상 응답 |
| Security | 승인되지 않은 관리자 접근 제한 |

## 8.3 Frontend 검증

| 구분 | 검증 항목 |
|---|---|
| Build | Vite 빌드 성공 |
| Auth | 로그인/로그아웃/라우터 가드 동작 |
| Dashboard | 차량 상태와 피로도 등급 표시 |
| Simulation | 시뮬레이션 실행/중지 UI 동작 |
| Responsive | 주요 화면에서 레이아웃 깨짐 여부 확인 |

## 8.4 AI 검증

| 구분 | 검증 항목 |

## 9. 결함 처리 기준

릴리즈 브랜치에서 발견된 결함은 심각도에 따라 처리한다.

| 심각도 | 기준 | 처리 |
|---|---|---|
| Critical | 실행 불가, 로그인 불가, DB 초기화 실패 | 릴리즈 중단 후 수정 |

## 10. 핫픽스 전략

`main`에 릴리즈가 반영된 뒤 제출 또는 발표 직전에 치명적인 문제가 발견되면 `hotfix/*` 브랜치를 사용한다.

### 핫픽스 흐름

```text
main → hotfix/v1.0.1-{issue} → main → tag → dev 반영
```

### 예시

```bash
git switch main
git pull origin main

git switch -c hotfix/v1.0.1-login-error
git push -u origin hotfix/v1.0.1-login-error
```

핫픽스가 `main`에 병합되면 반드시 `dev`에도 반영한다.

```text
main의 hotfix 반영 내용 → dev 병합 또는 cherry-pick
```

| `v0.1.0` | 프로젝트 개요서, 요구사항 정의서, DB 설계서, 프로젝트 구조도, WBS, 브랜치 전략, 릴리즈 전략 |

## 12. 현재 시점 적용안

2026.04.30 현재 프로젝트는 초기 기능 브랜치와 산출물 브랜치가 나뉘어 있는 상태다.

따라서 다음 목표 릴리즈는 `v0.1.0`이 적절하다.

### `v0.1.0` 전 dev 병합 대상

| 브랜치 | 상태 기준 |
|---|---|
| `feature/docs-deliverables` | 산출물 정리 완료 후 dev 병합 |
| `feature/fe-project-setup` | FE 초기 실행 확인 후 dev 병합 |
| `feature/be-entity-db` | Entity/DB 구조 확인 후 dev 병합 |
| `feature/infra-db-init` | init/seed 실행 확인 후 dev 병합 |
| `feature/sim-project-setup` | Python 기본 구조 확인 후 dev 병합 |

### `v0.1.0` 생성 전 확인

```text
1. 각 feature 브랜치 PR을 dev에 병합한다.
2. dev에서 프로젝트 구조와 문서를 확인한다.
3. release/v0.1.0 브랜치를 생성한다.
4. release/v0.1.0에서 실행/문서/환경변수를 검증한다.
5. release/v0.1.0을 main으로 PR한다.
6. main 병합 후 v0.1.0 태그를 생성한다.
```

## 13. 최종 정리

`logmile`의 릴리즈 전략은 다음 원칙을 따른다.

1. 기능 개발은 `feature/*`에서 진행한다.
2. 기능 통합은 `dev`에서 진행한다.
3. 마일스톤 검증은 `release/*`에서 진행한다.
4. 제출 가능한 안정 버전만 `main`에 반영한다.
5. `main`에 반영된 릴리즈는 반드시 태그를 생성한다.
6. 제출 버전은 `v1.0.0`, 발표 안정화 버전은 `v1.0.1`로 분리한다.
7. 릴리즈 브랜치에서는 신규 기능 추가보다 안정화와 검증을 우선한다.
