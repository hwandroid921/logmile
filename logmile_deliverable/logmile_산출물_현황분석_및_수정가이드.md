# logmile 산출물 현황 분석 및 수정 가이드

- 프로젝트명: `logmile`
- 문서 버전: `v1.2`
- 작성 기준일: `2026.05.15`
- 작성 기준: 실제 프로젝트 구현물, 핵심 산출물 `md`, 브랜치 전략, 릴리즈 전략

## 2. 현재 프로젝트 의도 재정의

현재 `logmile`의 실제 의도는 단순한 관제 대시보드 구현이 아니라, 아래 흐름을 시연 가능한 하나의 시스템으로 연결하는 것이다.

```text
업체/관리자 승인 기반 멀티테넌시
  → 차량/운전자 관리
  → 시뮬레이션 운행 생성
  → GPS 데이터 수신
  → 휴식/야간/연속 운행 기반 피로도 계산
  → 대시보드/운행 이력/통계 확인
  → 번호판 OCR 및 입출차 이벤트 보조
```

따라서 산출물은 기능 목록을 단순 나열하기보다, 위 흐름이 어떤 문서와 코드에서 어떻게 구현되는지 설명해야 한다.

## 3. 현재 수준 판단

| 영역 | 현재 수준 | 판단 |
|---|---|---|
| Backend | 인증, 승인, 업체, 차량, 운전자, GPS, 피로도, 대시보드, 이력, 통계 API 구현 | 핵심 도메인 구현은 상당 부분 완료 |
| Frontend | 로그인/회원가입/승인/업체관리/피로도통계 API 연동, 대시보드/이력/차량/운전자/임계값 화면 존재 | 일부 화면은 API 연동 완료, 주요 관제 화면에는 mock 데이터 기반이 남아 있음 |
| AI | FastAPI OCR 라우터와 YOLO/EasyOCR 서비스 구조 존재 | OCR API 구조는 있으나 실제 모델/데이터 검증 결과 정리가 필요 |
| Simulator | A/B/C 시나리오, GPS 전송, 번호판 이벤트 생성 기능 존재 | 시뮬레이션 구조는 구현됐고 통합 검증 문서화가 필요 |
| Infra | PostgreSQL, BE, AI, FE, Simulator Docker Compose 구성 | 통합 실행 기반은 있음 |
| Deliverables | 산출물 수와 범위는 충분함 | 버전, DB 구조, 권한 구조, API 경로, mock/API 상태 정합성 보완 필요 |

현재 프로젝트는 `v0.2.0`의 백엔드/AI 핵심 구현 범위를 상당 부분 달성했고, `v0.3.0`의 프론트엔드/시뮬레이터 통합 단계에 진입한 상태로 보는 것이 적절하다.

## 4. 산출물 전체 수정 방향

### 4.1 기준선 정리

현재 산출물은 초기 기획 문서와 구현 이후 문서가 함께 존재하면서 버전 기준이 흔들린다.

| 문제 | 수정 방향 |
|---|---|
| 산출물 기준정리표는 핵심 문서를 `v1.5` 기준으로 보지만, 실제 문서에는 `v4.0`, `v5.0`이 혼재 | `v1.7` 기준정리표에서 실제 운영 기준을 재정리 |
| `md`와 `docx` 기준이 섞여 보임 | `md`를 최신 원본, `docx`를 제출/보관본으로 재확인 |
| 문서별 작성 기준일이 현재 구현 상태와 다름 | 핵심 문서 상단에 최신 반영일과 변경 기준 추가 |

권장 기준:

```text
현재 구현 반영 산출물 기준: v5.2
작성 기준일: 2026.05.15
기준 원칙: md 우선, docx 요청 시 갱신
```

### 4.2 구현 상태 표현 방식

산출물에서는 “완료”와 “화면 존재”를 구분해야 한다.

| 표현 | 의미 |
|---|---|
| 구현 완료 | 실제 API 또는 기능이 동작하며 검증 가능한 상태 |
| 1차 구현 | 코드 구조와 주요 흐름은 있으나 통합 검증이 추가로 필요한 상태 |
| 화면 구현 | Vue 화면은 있으나 mock 데이터 또는 API 연동 보완이 필요한 상태 |
| 통합 필요 | BE/FE/SIM/AI 사이 계약 또는 실행 흐름이 아직 닫히지 않은 상태 |
| 시연용 대체 입력 | 실제 자동 수집 데이터를 시연 페이지에서 직접 입력해 기존 결과 테이블에 반영하는 상태 |

## 5.4 `logmile_DB설계서.md`

### 현재 문제

- 현재 실제 SQL에는 `company`, `plate_event`가 포함되어 있으나 DB 설계서 본문에는 과거 구조가 남아 있다.
- `admin`을 독립 테이블로 설명하는 내용은 현재 `company_id` 기반 구조와 맞지 않는다.
- `vehicle`, `driver`, `drive_log`의 `company_id`가 핵심 설계 요소로 충분히 반영되어 있지 않다.
- 시드 데이터 설명은 실제 `seed.sql` 기준과 일부 차이가 있다.

### 수정 방향

| 항목 | 수정 내용 |
|---|---|
| 엔티티 목록 | `company`, `plate_event` 추가 |
| ERD | `company → admin/vehicle/driver/drive_log` 관계 추가 |
| admin 테이블 | `company_id`, `status`, `ROLE_SUPER_ADMIN`, `ROLE_ADMIN` 반영 |
| vehicle/driver/drive_log | `company_id` 반영 |
| plate_event | 입출차 이벤트, 관측 위치, 입력 출처, confidence, manual 여부 반영 |
| 시드 데이터 | 회사 10개, 관리자 31명, 차량 50대, 운전자 50명, 임계값 21건으로 정리 |
| DDL 버전 | `init.sql`, `seed.sql`의 `v5.0`과 문서 기준 맞춤 |

## 5.6 `logmile_WBS_초안.md`

### 현재 문제

- 상태가 과거 일정 기준으로 남아 있어 실제 구현 진척과 맞지 않는다.
- 이미 구현된 BE 기능이 `예정` 또는 `진행`으로 남아 있다.
- FE 화면은 존재하지만 API 연동 상태가 분리되어 있지 않다.
- 과거 업체 관리자 권한명, 과거 번호판 이벤트 클래스명 등 실제 코드와 다른 용어가 남아 있었다.

### 수정 방향

| 항목 | 수정 내용 |

## 8. 피로도 계산 설명 보강 방향

피로도 계산은 프로젝트 핵심이므로 산출물에서 실제 구현 흐름을 명확히 설명해야 한다.

현재 구현 기준 흐름:

```text
POST /api/gps
  → GpsData 저장
  → RestEventService.process()
  → FatigueScoreService.calculate()
  → FatigueEvent 저장
  → DriveLog max fatigue 갱신
  → Dashboard polling에서 최신 상태 조회
```

문서에 명시할 전제:

| 항목 | 설명 |
|---|---|
| 연속 운행 | 마지막 유효 휴식 종료 시각 이후부터 현재 GPS 기록 시각까지 계산 |
| 일일 운행 | 운전자 기준 당일 GPS 기록 수와 시뮬레이터 간격을 기반으로 계산 |
| 야간 운행 | 22:00~06:00 구간 GPS 기록을 누적 |
| 휴식 판단 | `speed_kmh <= 3` 상태가 15분 이상이면 유효 휴식 |
| 충분 휴식 | 30분 이상 정차 시 충분 휴식 |
| 판단 근거 | `fatigue_event.reason`에 점수 산정 근거 저장 |
| 임계값 | `fatigue_threshold` 테이블은 존재하나 계산 로직과의 동적 연동 여부는 별도 명시 필요 |

## 10. 결론

현재 `logmile` 산출물은 양과 범위는 충분하지만, 초기 기획 기준과 실제 구현 기준이 한 폴더에 공존하면서 기준선이 흐려진 상태다.

따라서 지금의 산출물 수정 핵심은 다음 세 가지다.

1. 문서 버전과 최신 기준선을 정리한다.
2. 실제 구현된 `company`, `AdminStatus`, `PlateEvent`, API 경로, 멀티테넌시 흐름을 반영한다.
3. FE 화면의 mock 데이터 기반 상태와 실제 API 연동 상태를 분리해 관리한다.

이 정리가 끝나면 이후 코드 수정과 발표 준비는 `로그인 → 승인 → 시뮬레이션 → GPS → 피로도 → 대시보드 → 이력/통계`라는 하나의 검증 흐름을 기준으로 진행할 수 있다.
